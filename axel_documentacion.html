03 — Tools de Axel
fn
buscar_factura
Busca factura pendiente por WhatsApp del consumidor. Devuelve monto, vencimiento y concepto del servicio.
fn
validar_comprobante
Recibe imagen del comprobante de transferencia. Extrae monto y valida contra factura esperada. Llama a la API bancaria.
fn
emitir_recibo
Genera recibo de A Team y (en MVP) factura fiscal vía ARCA. Envía al consumidor por WhatsApp y email. Actualiza Cloud SQL.
fn
escalar_a_humano
Transfiere la conversación a un agente humano en Chatwoot con historial completo. Notifica al equipo A Team inmediatamente.
fn
enviar_cadencia
Dispara el siguiente mensaje proactivo de la cadencia configurada por cliente. Respeta ventanas horarias y estado.
fn
actualizar_estado
Actualiza el estado del consumidor en Cloud SQL y refleja en Dashboard en tiempo real.
fn
generar_link_pago
Genera link de pago personalizado on-the-fly con monto exacto y CVU/alias del cliente. Se incluye en cada solicitud de pago.
fn
registrar_promesa
Registra fecha de pago prometida. Programa recordatorio D-1 y mensaje día D en Cloud Scheduler. Marca conversación como snoozed en Chatwoot.

Decisiones de diseño
✗ Sin tabla links_pago
Links generados on-the-fly en runtime: factura.monto + cuentas_recaudadoras.cvu. No hay estado nuevo que persistir.
✓ consumidores_clientes necesaria
Un consumidor puede deber a múltiples clientes de A Team. El estado de cobranza vive en la join table por relación, no global.
✓ Promesas: modelo híbrido
Firestore guarda la promesa conversacional completa para Axel. Cloud SQL guarda solo el resumen (fecha, monto, estado) para el Dashboard.

05 — Workflow de ejemplo (deploy-demo.yml)
.github/workflows/deploy-demo.yml 
name: Deploy → Demo

on:
  push:
    branches: [demo]

env:
  GCP_PROJECT_ID: ${{ secrets.GCP_PROJECT_ID_DEMO }}
  GCP_REGION: us-east1
  REGISTRY: us-east1-docker.pkg.dev

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Auth GCP
        uses: google-github-actions/auth@v2
        with:
          credentials_json: ${{ secrets.GCP_SA_KEY_DEMO }}

      - name: Configure Docker
        run: gcloud auth configure-docker $REGISTRY

      - name: Build & Push receptor
        run: |
          docker build -t $REGISTRY/$GCP_PROJECT_ID/axel/receptor:${{ github.sha }} ./functions/receptor
          docker push $REGISTRY/$GCP_PROJECT_ID/axel/receptor:${{ github.sha }}

      - name: Build & Push procesador
        run: |
          docker build -t $REGISTRY/$GCP_PROJECT_ID/axel/procesador:${{ github.sha }} ./functions/procesador
          docker push $REGISTRY/$GCP_PROJECT_ID/axel/procesador:${{ github.sha }}

      - name: Deploy receptor a Cloud Run
        run: |
          gcloud run deploy axel-receptor-demo \
            --image $REGISTRY/$GCP_PROJECT_ID/axel/receptor:${{ github.sha }} \
            --region $GCP_REGION \
            --set-env-vars AMBIENTE=demo \
            --set-secrets CLAUDE_API_KEY=claude-api-key:latest \
            --allow-unauthenticated

      - name: Deploy procesador a Cloud Run
        run: |
          gcloud run deploy axel-procesador-demo \
            --image $REGISTRY/$GCP_PROJECT_ID/axel/procesador:${{ github.sha }} \
            --region $GCP_REGION \
            --set-env-vars AMBIENTE=demo \
            --set-secrets CLAUDE_API_KEY=claude-api-key:latest \
            --no-allow-unauthenticated

⚠ Reglas de secrets
Los secrets sensibles (API keys, tokens) van en GitHub Secrets y se inyectan como env vars en el workflow. Nunca en el código ni en el repo.
Las variables no sensibles (URLs, nombres de proyectos) pueden ir en GitHub Variables (sección Settings → Variables).
En GCP, los secrets se manejan con Google Secret Manager y se referencian en Cloud Run con --set-secrets.
07 — Cómo probar una branch antes de mergear
Opción A — Test local con .env.demo
Creás un .env.local con las variables de demo
cp .env.demo .env.local
Corrés el procesador local:
npm run dev --workspace=functions/procesador
Usás ngrok para exponer el receptor local:
ngrok http 3000
Apuntás el webhook de Chatwoot al túnel ngrok
Opción B — Deploy de branch a ambiente demo
Mergeás temporalmente tu branch a demo:
git checkout demo
git merge feature/mi-feature --no-ff
git push origin demo
GitHub Actions despliega en minutos
Probás con el número de demo en WhatsApp
Si falla, revertís: git revert HEAD
Opción C — Logs en tiempo real desde GCP
Para ver logs en tiempo real de cualquier ambiente:
gcloud run services logs read axel-procesador-demo \
  --region us-east1 --tail 50 --follow
O desde la consola GCP → Cloud Run → Logs
Filtrás por severity=ERROR para ver solo errores
Revertir un deploy roto en producción
Cloud Run guarda las revisiones anteriores.
Rollback inmediato sin tocar código:
gcloud run services update-traffic \
  axel-procesador-prod \
  --to-revisions PREV_REVISION=100
Mientras tanto, revertís en git y redesplegás.

Runbook de operaciones · v1.0

Runbook Operaciones
Qué hacer cuando algo falla

Diagnóstico · Recuperación · Rollback · Alertas

🔴 P1 — Axel no responde mensajes
Diagnóstico — en este orden
1. ¿Chatwoot recibe el mensaje?
   → Entrar al dashboard y verificar
2. ¿El webhook llega al receptor?
   gcloud run services logs tail axel-receptor-demo
3. ¿El mensaje se encola en Cloud Tasks?
   → GCP Console → Cloud Tasks
4. ¿El procesador falla?
   gcloud run services logs tail axel-procesador-demo
5. ¿Claude API devuelve error?
   → Buscar en logs: AnthropicError
Soluciones frecuentes
Cloud Run caído: reiniciar manualmente
gcloud run services update axel-procesador-demo --region us-east1
Claude API rate limit: verificar uso en console.anthropic.com · el procesador reintenta automáticamente
Lock de Firestore trabado: ver sección P3 más abajo
Webhook URL incorrecta: verificar en Chatwoot Settings → Integrations → Webhooks
🔴 P2 — Chatwoot caído o inaccesible
SSH a la VM y verificar el estado de los contenedores:
gcloud compute ssh chatwoot-vm --zone=us-east1-b
cd chatwoot && docker-compose ps
docker-compose logs --tail=50 chatwoot
Si algún contenedor está caído:
docker-compose restart chatwoot
Si la VM no responde → GCP Console → Compute Engine → reiniciar instancia.
⚠ Mientras Chatwoot está caído, los mensajes de WhatsApp se siguen recibiendo en Meta/Evolution API. Al volver, Chatwoot sincroniza automáticamente los mensajes perdidos.
🟡 P3 — Lock de Firestore trabado (conversación bloqueada)
Síntoma: un consumidor manda mensajes pero Axel no responde. En Firestore, el documento de la conversación tiene processing: true con un processing_since viejo.
Diagnóstico:
En Firestore Console → colección conversations → buscar por whatsapp del consumidor → ver campo processing.
Fix manual: editar el documento en Firestore Console y setear processing: false.
El procesador tiene detección automática de locks caídos: si processing_since tiene más de 5 minutos, libera el lock automáticamente. Si el problema persiste, el lock manual es la solución inmediata.
🔴 P4 — Deploy roto en producción → rollback
Rollback inmediato sin tocar código (segundos):
gcloud run revisions list --service axel-procesador-prod --region us-east1
gcloud run services update-traffic axel-procesador-prod \
  --to-revisions axel-procesador-prod-00023-abc=100 --region us-east1
Repetir para el receptor si también está afectado.
Luego: revertir en git → push a staging → validar → push a main → deploy con aprobación.
🟡 P5 — Cloud SQL lento o con errores de conexión
Ver métricas en GCP Console → Cloud SQL → tu instancia → Monitoring.
Si hay demasiadas conexiones abiertas (Cloud Run escala y abre conexiones nuevas):
→ Usar Cloud SQL Auth Proxy o pg-pool en el procesador para limitar el pool de conexiones.
Reiniciar la instancia como último recurso:
gcloud sql instances restart axel-db-prod
⚠ El reinicio tarda ~60 segundos. Durante ese tiempo los procesadores van a fallar y Cloud Tasks va a reintentar automáticamente con backoff.
🟡 P6 — Evolution API desconectada (demo)
Evolution API puede desconectarse si WhatsApp Web detecta la sesión como inactiva o sospechosa.
Verificar el estado de la instancia:
curl http://localhost:8080/instance/connectionState/axel-demo \
  -H "apikey: tu-api-key"
Si devuelve state: "close", hay que volver a escanear el QR:
curl http://localhost:8080/instance/connect/axel-demo -H "apikey: tu-api-key"
Abrir la URL del QR en el browser y escanear con el teléfono de demo.
⚠ Este riesgo no existe en producción con Meta Cloud API oficial.
